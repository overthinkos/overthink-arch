# Changelog — 2026-07

### 2026-07-01 — refactor: read the welded-plugin set from the shared pkg/host-command-plugins.txt (R3)

The welded command plugin list (`plugin-secrets`/`udev`/`tmux`/`preempt`/`feature`/`vm`/
`doctor`/`clean`/`settings`/`candy`) was hardcoded in both `build()` and `package()`.
`build()` now reads it from the superproject's shared single source
`pkg/host-command-plugins.txt` — the SAME list the fedora `.rpm` and debian `.deb`
`local_pkg.build_template`s read, so the three package builders can never drift (R3).
`package()` now installs whatever `build()` staged (globs the staged `${srcdir}/*.providers`
set — the single build→package handoff), so the list lives in exactly one place.
No behavior change to the produced package; the same 10 plugins land at
`/usr/lib/charly/plugins`.

### 2026-07-01 — build: bake plugin-clean / plugin-settings / plugin-candy (C15 command externalizations)

`charly clean`, `charly settings`, `charly candy` were externalized to out-of-tree `command:<word>`
plugins (welded raw-forward to hidden `charly __clean`/`__settings`/`__candy`; the Run handlers stay
core). PKGBUILD now builds + bakes those three beside charly at the FHS plugin dir (both the build loop
and the package/`.providers` loop), so a packaged charly resolves them project-lessly. Roster is now:
plugin-secrets / plugin-udev / plugin-tmux / plugin-preempt / plugin-feature / plugin-vm / plugin-doctor /
plugin-clean / plugin-settings / plugin-candy.

**`charly version` is deliberately NOT externalized**: `pkgver()` stamps the package version via
`bin/charly version`, so `version` must resolve on a project-less/plugin-less dev binary — externalizing it
is an unfixable chicken-and-egg (RDD-vindicated). `version` stays a core command.

Verified: `task build:charly` installs system-wide with plugin-clean/settings/candy + `.providers` present
under `/usr/lib/charly/plugins`; `charly version` still works (2026.182.0002); the `check-commands-local`
disposable bed PASS (steps=4).

### 2026-07-01 — build: bake candy/plugin-doctor (command:doctor) into /usr/lib/charly/plugins

The `charly doctor` command was externalized to the out-of-tree `candy/plugin-doctor` (the fifth
welded-command externalization). PKGBUILD now builds + bakes `plugin-doctor` beside charly at the FHS
plugin dir (both the build loop and the package/`.providers`-manifest loop), so a packaged charly resolves
`charly doctor` project-lessly (discoverBakedPluginWords → `command:doctor` → syscall.Exec → the hidden
in-core `charly __doctor`). The bundled-plugin roster is now:
plugin-secrets / plugin-udev / plugin-tmux / plugin-preempt / plugin-feature / plugin-vm / plugin-doctor.

Verified: `task build:charly` installs system-wide with `plugin-doctor` + `plugin-doctor.providers` present
under `/usr/lib/charly/plugins`; the `check-doctor-local` disposable bed PASS (steps=4) exercises the baked
command end-to-end.
